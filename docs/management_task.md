# タスク管理憲章

## 1. 概要

本憲章は、プロジェクトにおけるタスク管理の原則と手法を定義し、効率的で透明性の高い開発プロセスを実現することを目的とします。

## 2. タスク管理の階層構造

### 2.1 階層の定義

```
docs/todos/            # フェーズレベルのタスク（大規模）
└── docs/dev/         # 実装レベルのタスク（小規模）
    └── phase-XX/     # 各フェーズの詳細タスク
```

### 2.2 各階層の役割

#### docs/todos/ - 戦略レベル
- **視点**: プロダクトオーナー、プロジェクトマネージャー
- **粒度**: 2-4週間単位の大きな機能やマイルストーン
- **内容**: ビジネス価値、ユーザーストーリー、定義完了条件
- **例**: "認証システムの実装", "テンプレート機能の構築"

#### docs/dev/phase-XX/ - 戦術レベル
- **視点**: 開発者、テクニカルリード
- **粒度**: 数時間〜1日で完了可能な具体的タスク
- **内容**: 技術的実装詳細、TDDサイクル、具体的な手順
- **例**: "AuthManagerインターフェースの定義", "jest.config.jsの作成"

## 3. TDD（テスト駆動開発）の原則

### 3.1 Red-Green-Refactorサイクル

各タスクは必ず以下の3フェーズで構成されます：

#### RED フェーズ（失敗するテストを書く）
- **目的**: 実装前に期待される動作を明確化
- **時間**: 15-30分
- **成果物**: 失敗するテストコード
- **チェックポイント**:
  - [ ] テストが期待通りに失敗するか
  - [ ] テストが要件を正確に表現しているか
  - [ ] エラーメッセージが明確か

#### GREEN フェーズ（テストを通す）
- **目的**: 最小限の実装でテストを成功させる
- **時間**: 30-60分
- **成果物**: テストが通る実装コード
- **チェックポイント**:
  - [ ] 全てのテストが成功するか
  - [ ] 過剰な実装をしていないか
  - [ ] 既存のテストが壊れていないか

#### REFACTOR フェーズ（コードを改善する）
- **目的**: コードの品質向上と保守性の確保
- **時間**: 15-45分
- **成果物**: リファクタリングされたコード
- **チェックポイント**:
  - [ ] テストが継続して成功するか
  - [ ] コードの重複が除去されているか
  - [ ] 命名規則が統一されているか
  - [ ] ドキュメントが更新されているか

### 3.2 TDDの利点
- 設計の明確化
- 回帰テストの自動化
- リファクタリングの安全性
- ドキュメントとしてのテスト

## 4. タスクサイズの原則

### 4.1 理想的なタスクサイズ

#### マイクロタスク（推奨）
- **所要時間**: 1-2時間
- **TDDサイクル**: 1回完結
- **例**: "エラークラスの作成", "設定ファイルの検証"
- **利点**: 
  - 高い完了率
  - 明確な進捗可視化
  - 低い認知負荷

#### スモールタスク
- **所要時間**: 2-4時間
- **TDDサイクル**: 1-2回
- **例**: "CLIコマンドの実装", "設定ジェネレーターの作成"
- **注意点**: 必要に応じてサブタスクに分割

#### ミディアムタスク（避けるべき）
- **所要時間**: 4時間以上
- **問題点**: 
  - 完了の定義が曖昧
  - 中断リスクが高い
  - 見積もりの精度が低下

### 4.2 タスク分割の基準

タスクは以下の基準で分割します：

1. **単一責任の原則**: 1つのタスクは1つの責任のみ
2. **テスタビリティ**: 独立してテスト可能
3. **依存関係の最小化**: 他タスクへの依存を最小限に
4. **明確な完了条件**: 客観的に判断可能な完了基準

## 5. タスクファイルの構造

### 5.1 標準フォーマット

```markdown
# Task XXX: [タスク名]

## Goal
[このタスクで達成すべき目標を1-2文で記述]

## Tasks

### XXX-1: RED - [テストの説明]
- [ ] [具体的なテスト項目1]
- [ ] [具体的なテスト項目2]
- [ ] テストを実行して失敗することを確認

### XXX-2: GREEN - [実装の説明]
- [ ] [具体的な実装項目1]
- [ ] [具体的な実装項目2]
- [ ] テストを実行して成功することを確認

### XXX-3: REFACTOR - [改善の説明]
- [ ] [具体的な改善項目1]
- [ ] [具体的な改善項目2]
- [ ] テストが継続して成功することを確認

## Definition of Done
- [完了条件1]
- [完了条件2]
- テストが全て通っている
```

### 5.2 チェックボックスの運用

- `- [ ]`: 未着手
- `- [x]`: 完了
- 各フェーズ完了時に即座に更新
- 進捗の可視化と追跡に使用

## 6. 実装例

### 6.1 良いタスク分割の例

```
Phase 1: Core Implementation (docs/todos/)
└── Task 001: Setup Project Structure (docs/dev/phase-01/)
    ├── 001-1: RED - プロジェクト構造のテストを書く
    ├── 001-2: GREEN - 基本的なプロジェクト構造を作成
    └── 001-3: REFACTOR - プロジェクト構造の改善
```

### 6.2 悪いタスク分割の例

```
❌ Task: 認証システムを実装する
理由: 大きすぎる、複数の責任、テスト不可能
```

## 7. ベストプラクティス

### 7.1 タスク作成時
1. ユーザーストーリーから始める
2. 技術的タスクに分解する
3. 各タスクにTDDサイクルを適用
4. 依存関係を明確化

### 7.2 タスク実行時
1. 1つのタスクに集中（WIP制限）
2. 各フェーズ完了時に即座にマーク
3. ブロッカーは新規タスクとして記録
4. 定期的な進捗レポートの更新

### 7.3 タスク完了時
1. Definition of Doneの確認
2. ドキュメントの更新
3. 次のタスクの準備
4. 知見の記録と共有

## 8. メトリクスと改善

### 8.1 追跡すべきメトリクス
- タスク完了率
- サイクルタイム（着手から完了まで）
- テストカバレッジ
- リワーク率

### 8.2 継続的改善
- 週次でのタスクサイズレビュー
- 月次でのプロセス振り返り
- 四半期ごとの憲章見直し

## 9. まとめ

本憲章は、以下の価値を重視します：

1. **透明性**: 進捗が誰にでも見える
2. **予測可能性**: 小さなタスクによる安定した進捗
3. **品質**: TDDによる堅牢な実装
4. **持続可能性**: 無理のないペースでの開発

これらの原則に従うことで、チーム全体の生産性と満足度を向上させることができます。
